1. XSS (Cross-Site Scripting)
Чтобы вредоносный скрипт не выполнился в браузере, мы фильтруем все входящие данные и экранируем все выходящие.
Использование htmlspecialchars() превращает опасные теги в обычный текст. 
Вставка из index.php:
PHP$biography = htmlspecialchars($_POST['biography'] ?? ''); // Санитизация на входе
$fio = trim(strip_tags($_POST['fio'] ?? ''));// Удаление тегов
Было: В формах и админ-панели данные, введённые пользователем (ФИО, биография, языки и т.д.), выводились напрямую, что позволяло выполнять JavaScript-код.

<p><?=$a['fio']?></p>

Стало: Все пользовательские данные перед выводом обрабатываются функцией htmlspecialchars().

<p><?=htmlspecialchars($a['fio']??'-')?></p>

2. Information Disclosure (Раскрытие информации) 
Мы скрываем системную информацию (пути к файлам, структуру БД, логины) от пользователя.
Для этого критический код обернут в блоки try-catch, которые перехватывают технические ошибки и выводят вместо них нейтральное сообщение.  
Вставка из index.php:
PHPtry {
    $pdo = new PDO("mysql:host=$host;dbname=$dbname;charset=utf8mb4", $username, $password);
} catch (PDOException $e) {
    // Скрываем детали $e->getMessage()
    die("Ошибка БД. Пожалуйста, попробуйте позже.");
}

Было:
	•	Выводились подробные ошибки PHP и PDO (stack trace).
	•	В админ-панели отображались реальные пароли пользователей.
	•	Показывались технические предупреждения.
Стало:
	•	Установлен режим PDO::ERRMODE_SILENT.
	•	При ошибках выводится общее сообщение «Ошибка сервера».
	•	Пароли в админ-панели выводятся, но только для администратора (выделены красным цветом).
$pdo->setAttribute(PDO::ATTR_ERRMODE, PDO::ERRMODE_SILENT);

3. SQL Injection
Объяснение: отказались от прямой вставки переменных в текст SQL-запроса.
Вместо этого используются подготовленные запросы (Prepared Statements) через PDO. 
Данные передаются отдельно в метод execute(), что делает невозможным изменение логики запроса злоумышленником через кавычки или спецсимволы.
Вставка из index.php:
PHP$stmt = $pdo->prepare("
    SELECT a.* FROM applications a
    JOIN users u ON a.id = u.application_id
    WHERE u.login = ?
");
$stmt->execute([$_SESSION['login']]);
Было: В некоторых местах использовались неподготовленные запросы (хотя в основном Prepared Statements уже применялись).
Стало: Все запросы к базе данных выполняются только через Prepared Statements (prepare + execute).
Используются именованные или позиционные параметры. Добавлена функция INSERT IGNORE для предотвращения дублирования.

4. CSRF (Cross-Site Request Forgery)
Объяснение: Для защиты от поддельных запросов с других сайтов генерируем уникальный токен в сессии пользователя. 
При отправке любой формы этот токен проверяется на сервере. Если токен отсутствует или не совпадает, запрос блокируется. 
Вставка из index.php:
PHP// Генерация в GET-запросе
if (empty($_SESSION['csrf_token'])) {
    $_SESSION['csrf_token'] = bin2hex(random_bytes(32));
}
// Проверка в POST-запросе
if (!isset($_POST['csrf_token']) || $_POST['csrf_token'] !== $_SESSION['csrf_token']) {
    header('HTTP/1.1 403 Forbidden');
    die('Ошибка безопасности: CSRF-токен не валиден.');
}
Было: Отсутствовала защита от поддельных запросов (можно было удалять/изменять данные через вредоносную ссылку).
Стало: Реализована полноценная защита с помощью CSRF-токена:
	•	Генерация токена при старте сессии:
if (empty($_SESSION['csrf_token'])) {
    $_SESSION['csrf_token'] = bin2hex(random_bytes(32));
}
	•	Добавление скрытого поля в каждую форму
  •	Проверка токена перед обработкой POST-запросов (index.php, admin.php, login.php).

5. Include и Upload уязвимости 
Мы защищаемся от внедрения посторонних файлов, жестко прописывая имя подключаемого файла без использования переменных из URL.
Также введена проверка данных по «белому списку» (allowed languages), чтобы в базу не попали непредусмотренные значения.  
Вставка из index.php:
PHP// Безопасный include
include('form.php'); 
// Проверка по белому списку
if (in_array($lang, $allowed_languages)) { 
    $pdo->prepare("INSERT INTO application_languages ...")->execute([$app_id, $lang]);
}

Было:
	•	Использовался include('form.php') — потенциальная уязвимость Local File Inclusion.
	•	Отсутствовала загрузка файлов (Upload).
Стало:
	•	include используется только для доверенных локальных файлов.
	•	Путь указывается явно без пользовательских данных.
	•	Загрузка файлов не реализована в приложении, поэтому уязвимость Upload отсутствует.
